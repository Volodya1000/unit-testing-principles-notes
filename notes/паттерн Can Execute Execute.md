## 🛑 Проблема: Конфликт производительности и инкапсуляции

При стремлении сделать контроллеры «тонкими», мы часто выносим все чтение данных (I/O) в начало метода контроллера, а саму операцию передаем в модель.
Однако иногда для выполнения операции требуются «тяжелые» данные (например, загрузка связанных сущностей из БД).

Возникает дилемма:
1. **Вариант "Слепое чтение":** Мы *всегда* загружаем тяжелые данные, передаем их в метод модели, а модель внутри проверяет, можно ли выполнить действие. Если нельзя — мы зря нагрузили базу данных.
2. **Вариант "Дырявая инкапсуляция":** Чтобы не грузить лишнее, мы ставим `if` в контроллере (`if (!user.IsConfirmed) ...`) т.е. проверка состояния доменной сущности прямо  в контроллере, для того чтоб решить нужно ли грузить ещё данные из БД или другая ветвь  выполнения контроллера.
 **Минусы:** 
 - Бизнес-логика (правило подтверждения) утекает из Домена в Контроллер.
 - Модель перестает отвечать за свои инварианты (можно вызвать метод изменения, минуя проверку).
 - Тесты доменной сущности теперь НЕ покрывают полностью защиту от багов.

## 💡 Решение: Паттерн CanExecute/Execute

Паттерн предлагает разбить бизнес-операцию на два этапа внутри **Доменной модели**:

1.  **CanExecute**: Метод-валидатор. Проверяет *только* состояние самой сущности. Возвращает `true/false`. Не имеет побочных эффектов.
2.  **Execute**: Метод действия. Выполняет бизнес-логику. **Обязательно** должен вызывать `CanExecute`как предусловие (Precondition).

### Алгоритм работы контроллера:
1. Загружаем минимально необходимые данные (агрегат).
2. Вызываем `CanExecute`.
3. Если **Ошибка** -> возвращаем ответ клиенту (не делаем тяжелых запросов).
4. Если **ОК** -> подгружаем остальные данные (если надо) и вызываем `Execute`.

## 💻 Пример кода

**Доменная модель (User):**
```csharp
public class User
{
    public bool IsEmailConfirmed { get; private set; }

    // 1. Проверка возможности (CanExecute)
    public string CanChangeEmail()
    {
        if (IsEmailConfirmed)
            return "Can't change a confirmed email";
        return null; // или возвращаем Success
    }

    // 2. Исполнение (Execute)
    public void ChangeEmail(string newEmail, Company company)
    {
        // Защита инварианта: нельзя выполнить без проверки
        Precondition.Requires(CanChangeEmail() == null); 
        
        // Логика изменения...
    }
}
```

**Контроллер:**
```csharp
public string ChangeEmail(int userId, string newEmail)
{
    // 1. Грузим только User
    User user = _repo.GetUser(userId);

    // 2. Проверяем "Fail Fast"
    string error = user.CanChangeEmail();
    if (error != null)
        return error; // Возврат без лишних запросов к БД

    // 3. Только если можно - грузим тяжелые зависимости
    Company company = _repo.GetCompany();
    
    // 4. Выполняем действие
    user.ChangeEmail(newEmail, company);
    
    _repo.Save(user);
    return "OK";
}
```

## ✅ Преимущества паттерна

1.  **Сохранение инварианта:** Доменная модель защищена. Нельзя вызвать `ChangeEmail` в обход правил, так как внутри стоит `Precondition`.
2.  **Чистый контроллер:** Контроллер не принимает бизнес-решений, он лишь оркестрирует поток выполнения на основе ответов модели.
3.  **Производительность:** Мы не выполняем дорогие операции I/O, если бизнес-правила запрещают действие на раннем этапе.
4.  **Тестируемость:** Всю сложность валидации можно покрыть юнит-тестами класса `User`. Контроллер остается тривиальным (Humble Object) и не требует сложных тестов.


## Ограничения паттерна
В некоторых ситуациях фрагментация бизнес-логики неизбежна. И данный паттерн не поможет.

Например:
- не существует способа проверки уникальности имейла за пределами контроллера без внедрения внепроцессных зависимостей в доменную модель. 
- сбой внепроцессной зависимости, который должен изменить ход бизнес операции. Решение относительно того, по какому пути пойти, не может находиться на уровне предметной области, потому что эти внепроцессные зависимости вызываются не слоем предметной области. Вам придется разместить эту логику в контроллере, а затем покрыть ее интеграционными тестами
