# Глава 9. Рекомендации при работе с моками

## 9.1. Максимизация эффективности моков

> [!warning] Совет
> При использовании моков всегда руководствуйтесь следующим принципом: проверяйте взаимодействия с неуправляемыми зависимостями на самых границах вашей системы.

В главе аргументируется что это нормально и правильно иногда делать несколько интерфейсов с наследованием. Пример:
- `IBus` и `IMessageBus`
- `ILogger` и `IDomainLogger`

Суть в том, что один из них реализует более низко уровневую логику, а другой более высоко уровневую. 
Но это две разные логики и поэтому их не нужно объединять в один интерфейс.

Например `IBus` отвечает за подключение к конкретной шине сообщений, настройку работы с ней, а `IMessageBus` за отправку конкретных типов сообщений.

> [!quote] Цитата
> Эти две обязанности — сокрытие сложности внешней библиотеки и хранение всех сообщений приложения в одном месте — лучше держать раздельно.
> 
> `IDomainLogger` реализует специфическую функциональность, требуемую бизнесом, для чего во внутренней реализации он использует обобщенный интерфейс `ILogger`.

*Рис. IBus находится на границе системы; IMessageBus — всего лишь промежуточное звено в цепочке типов между контроллером и шиной сообщений. Мокирование IBus вместо IMessageBus обеспечивает лучшую защиту от багов Потому что защита от регрессий зависит от объема кода, выполняемого в ходе теста*
![[Pasted image 20260126174349.png]]

Мокирование последнего типа, взаимодействующего с неуправляемой зависимостью, повышает количество классов, отрабатываемых интеграционным тестом, а следовательно, улучшает защиту.

> [!SUCCESS] Вывод
> Если в интеграционном тесте используется класс, который имеет интерфейс, но при этом мы не заменяем его моком, то можно использовать конкретный класс, а не интерфейс.

## 9.2. Замена моков шпионами
Как говорилось в главе 5 [[Принципы Unit тестирования Хориков Глава 5 Моки и хрупкость тестов]].

> [!abstract] Определение: Шпион (spy)
> **Шпион** представляет собой разновидность тестовой заглушки, которая служит той же цели, что и мок. Единственное различие заключается в том, что шпионы пишутся вручную, а моки создаются с помощью мок-фреймворков.

Для классов, находящихся возле границ системы, шпионы предпочтительнее моков. 
Шпионы помогают переиспользовать код в фазе проверок, сокращая тем самым размер и улучшая читаемость тесто.

Можно писать шпионов используя fluent (текучий) интерфейс.
Прямо внутри шпионов можно использовать assert.

**Пример шпиона:**
```csharp
public interface IBus
{
    void Send(string message);
}

public class BusSpy : IBus
{
    private readonly List<string> _sentMessages = new();

    public void Send(string message)
    {
        _sentMessages.Add(message);
    }

    public BusSpy ShouldSendNumberOfMessages(int number)
    {
        Assert.Equal(number, _sentMessages.Count);
        return this;
    }

    public BusSpy WithEmailChangedMessage(int userId, string newEmail)
    {
        string message = $"Type: USER EMAIL CHANGED; Id: {userId}; NewEmail: {newEmail}";
        Assert.Contains(_sentMessages, x => x == message);
        return this;
    }
}
```

**Оправданность писать шпиона появляется тогда, когда:**
1. Взаимодействие с зависимостью подразумевает передачу сложных данных (например, длинных форматированных строк, JSON-ов, бинарных данных), которые неудобно писать внутри `Verify(...)`.
2. Этот тип взаимодействия проверяется во многих тестах. То есть нарушается принцип DRY.
3. Вам нужно проверить именно **финальный результат** (сообщение, уходящее наружу), но вы хотите делать это удобно.

### Нужно ли заменять мокирование IDomainLogger на мокирование ILogger?
Автор считает что не нужно.
Точная структура текстовых логов не настолько важна для предполагаемой аудитории.
Важно само существование этих логов и хранящейся в них информации.

> [!SUCCESS] Вывод по прочитанному
> Были описаны две основные практики мокирования:
> - применение моков только к неуправляемым зависимостям;
> - проверка взаимодействий с этими зависимостями на границах системы.

## 9.3. Практики мокирования

**Основные правила:**
- применение моков только к неуправляемым зависимостям;
- проверка взаимодействий с этими зависимостями на границах системы (с самым общим интерфейсом);
- моки должны использоваться только в интеграционных, но не в юнит-тестах;
- проверка количества обращений к моку;
- мокирование только тех типов, которые вам принадлежат.

> [!fail] Ошибочная идея
> Идея что один интеграционный тест должен включать максимум один мок ошибочная. Так как интеграционный тест тестирует единицу поведения, а она может требовать несколько моков.

Количество моков зависит исключительно от количества неуправляемых зависимостей, участвующих в операции.

> [!important] Важно
> Важно проверять оба следующих аспекта:
> - наличие ожидаемых обращений к моку;
> - отсутствие неожиданных обращений к моку.
> 
> Это требование также происходит от необходимости обеспечения обратной совместимости с неуправляемыми зависимостями.

В большинстве мок-библиотек можно также явно проверить, что никакие другие вызовы к моку не совершаются. 
Функция типа `mock.VerifyNoOtherCalls()`.

> [!warning] Правило
> Вы всегда должны писать собственные адаптеры на базе сторонних библиотек и мокировать эти адаптеры вместо нижележащих типов.

**Аргументы:**
- внешний код может быть сложенный и трудный для понимания;
- даже если код уже предоставляет встроенные интерфейсы, мокирование этих интерфейсов часто сопряжено с риском, потому что вы должны быть уверены в том, что поведение мока совпадает с тем, что фактически делает внешняя библиотека;
- адаптеры абстрагируют несущественные технические детали стороннего кода и определяют отношения с библиотекой, используя доменный язык вашего приложения.

> [!NOTE] Мысль: Личный опыт
> По-моему пример такого адаптера это класс репозиторий обёртка над ef core или SQL alchemy.

**Адаптеры помогают:**
- абстрагировать сложность используемой библиотеки
- делать публичной только нужную вам функциональность из этой библиотеки 
- делать это с использованием языка предметной области вашего проекта.

Вы никогда не знаете, как изменится сторонний код при обновлении библиотеки. Такое обновление может привести к множеству ошибок компиляции по всему коду проекта. Дополнительный уровень абстракции ограничивает эти ошибки только одним классом: самим адаптером.
